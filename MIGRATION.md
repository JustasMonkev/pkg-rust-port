# pkg Rust Port Migration Map

This document is the source map for porting `vercel/pkg` into idiomatic Rust under `./rust-port`.
The JS test suite remains the behavioral oracle until Rust parity is complete.

## Goals

- Preserve the `pkg` CLI UX: package a Node.js project into a single executable-like artifact for selected Node/platform/arch targets.
- Port behavior leaf-first: path utilities, target parsing, source detection, dependency walking, VFS packing, payload production, then CLI orchestration.
- Keep the Rust API typed: enums for stores/targets/compression, newtypes for snapshot and host paths, explicit error types with `thiserror`, and `Result<T, E>` throughout.
- Keep JS fixtures outside the Rust crate and reference them as test data from the parent repository.

Initial Rust parity order:

1. `common`, `compress`, `target`, `config`.
2. `detect` against `test-50-ast-parsing*`, `test-50-require-*`.

4. `refine`, `pack`, and `produce` against VFS/compression fixtures.
5. CLI smoke tests mirroring the JS happy path.

## Decisions

- Use a single crate first. The JS package is one CLI/library package, and premature workspace splitting would add friction before parity.
- Keep sync library APIs by default. Network fetch, child process bytecode fabrication, and final production may expose async wrappers because they are I/O-bound.
- Convert dictionary JS modules into inert typed data. Executing JS dictionary modules from Rust would violate the no-vendored-JS-source direction and keep dynamic mutation in the design.
- Keep deterministic walker order. The JS code explicitly forbids multiple workers because task order affects output; Rust will use `VecDeque` FIFO and test this.
- Bound walker directory-link expansion to the entry tree by default. The JS walker can recurse through host parent links, but that makes Rust parity records depend on the developer machine outside the package under test.
- Separate walker root from snapshot base for package-directory inputs. The bounded Rust walker avoids enumerating host siblings, then refinement synthesizes the `/snapshot` directory record and keeps the package directory name in snapshot paths so runtime `realpath` and parent-directory reads match the JS suite.
- Track whether a package marker is top-level. JS stores JavaScript entries from top-level package `files` as blobs, but stores JavaScript entries from dependency package `files` as content so source text remains visible at runtime; Rust models this as `Marker::is_top_level()`.
- Treat a file input whose immediate parent has `package.json` as a package-scoped input for snapshot-base calculation. This preserves the package directory in `/snapshot` for fixtures like `test-50-package-json-6c` without letting unrelated ancestor package roots change ordinary single-file packaging.
- Discover `package.json` markers for local packages as well as `node_modules` packages, but bound that discovery to the active walker root. This includes local package metadata needed for runtime package-main resolution while preventing host repository package files from leaking into unrelated walks.
- Preserve the first `node_modules` segment when snapshotting direct file inputs from installed packages. JS denominator logic strips only the path before `node_modules`; Rust now mirrors that shape and synthesizes the intermediate snapshot directory links required by the prelude resolver.
- Treat dependency-derived resolver failures as non-fatal. JS emits warnings for dependency aliases that cannot resolve to a runtime entrypoint, which is required for metadata-only packages such as `@types/*`; Rust still follows the companion `dependency/package.json` path to activate transitive dependency metadata.
- Treat direct package resolution to an existing dependency `package.json` without `main` as a non-fatal warning. Rust records this as `PackageWarning::MissingMainEntry`, queues the package metadata, and lets the CLI render the JS-style warning on stdout.
- Add dictionary entries only as typed data when fixtures prove they are needed. `busboy` and `log4js` now map to the same script globs as their JS dictionary modules without executing vendored JS dictionary code.
- Preserve dictionary selection semantics. `--no-dict` disables built-in dictionary filenames such as `busboy.js`, `*` disables every dictionary entry, and root `pkg.dictionary` overrides load as typed package-specific dictionary entries.
- Convert dictionary log callbacks into typed `DictionaryLog` values. `stylus` now carries its asset glob and import-resolution warning as inert data; the walker turns that into a `PackageWarning` when package metadata activates.
- Keep dictionary deploy-file metadata in the same typed config shape as package `pkg.deployFiles`. `open` and its `opn` alias now carry the JS `xdg-open` patch and deploy-file warning data without executing the dictionary module. Directory deploy dictionaries such as `leveldown`, `puppeteer`, and `zeromq` preserve the third `deployFiles` kind field as typed warning data.
- Treat `pkg.deployFiles` as external distribution warnings at the walker boundary. These files are not embedded in the VFS; callers and npm fixture harnesses remain responsible for copying them beside the produced executable.
- Parse package source as CommonJS-compatible script input by allowing top-level `return`, matching the JS detector's handling of files that run inside Node's module wrapper.
- Traverse JavaScript detector nodes in breadth-first order, matching the JS walker's queue-style detector traversal. This keeps top-level malformed/non-literal diagnostics ahead of nested `try` diagnostics for `test-50-may-exclude-must-exclude`.
- Keep source diagnostics typed until the CLI boundary. Dynamic require, malformed require, optional missing-module, missing-main, and dictionary-log diagnostics all flow through `PackageWarning`; the CLI renders warning/debug prefixes and hides debug-only entries unless `--debug` is set.
- Traverse assignment expression right-hand sides during dependency detection. Spawn fixtures place `require.resolve(...)` inside assignments to child-process variables, and those child entrypoints must be bundled for runtime parity.
- Fall back from a forced file-entry snapshot base to the common path denominator when walked records escape that base. Native-addon fixtures can resolve sibling `node_modules` entries from an entrypoint under `lib/`, and slicing those escaped paths against `lib/` creates invalid snapshot keys.
- Synthesize POSIX parent directory records during common-denominator refinement. Bounded walking can discover escaped files without their parent directory links; the runtime prelude still needs those links for module traversal.
- Serialize stat payloads with the JS prelude field names (`isFileValue`, `isDirectoryValue`, `isSocketValue`, `isSymbolicLinkValue`) while keeping Rust's `FileStat` fields idiomatic with serde renames.
- Run CLI packaging on a dedicated larger-stack OS thread. This keeps `reqwest::blocking` outside the Tokio runtime and avoids stack overflow on larger VFS/prelude builds without requiring users to set `RUST_MIN_STACK`.
- Render fatal CLI packaging errors to stdout with the `> Error!` marker and no ANSI escapes. This preserves the JS invalid-fixture contract even though Rust library callers still receive typed `PkgError` values.
- Map CLI input and package-bin metadata misses to JS-style "does not exist" messages at the planning boundary. Lower-level filesystem operations keep structured `Io` errors, but user-facing invalid fixtures assert the original CLI wording.
- Keep the `test-46` output-name matrix covered at the planning layer. The JS suite mostly proves these fixtures by checking produced filenames, so Rust locks the same basename, extension stripping, explicit-output host fallback, single-target and multi-target suffix behavior, `--out-path`, `pkg.outputPath`, `pkg.targets`, scoped package basenames, and overwrite refusal before target binary production.
- Keep Node bytecode fabrication as an external process interaction. Rust cannot produce V8 cached data directly without a major V8 embedder dependency, so `fabricate` remains process-based.
- Carry `--build` as target metadata and require built cache artifacts for those targets. Real source-build orchestration remains separate; using fetched binaries for force-build requests would hide an important JS behavior boundary.
- Implement Node resolution directly. A subprocess bridge to Node's `resolve` package would be faster to write but would retain the JS resolver as part of the Rust product.

1. Cargo skeleton, CI, MSRV, lint policy, empty public API docs.
