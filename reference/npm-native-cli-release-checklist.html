Native npm CLI release checklist

Use this when reviewing packages like @jm-pkg-rust/pkg-rust, where a JavaScript launcher delegates to platform-specific native packages.

Mental model: the root package is only the switchboard. The native packages contain the executable payload. The release workflow is where those payloads are assembled.

Manifest checks

  • Root package.json has name, version, bin, files, engines.node, repository, license, homepage, and keywords.
  • files is an allowlist and excludes target/, test fixtures, and release build output.
  • Every native package in optionalDependencies has a matching npm/*/package.json.
  • Native package versions match Cargo.toml and the root package version.
  • Native packages declare the right os, cpu, and, for Linux, libc.

Launcher checks

  • The launcher maps every supported platform/architecture to exactly one native package.
  • Linux libc selection is explicit: glibc package for glibc, musl package for musl.
  • The launcher forwards args and preserves exit status.
  • Tests avoid Unix-only assumptions. On Windows, do not test direct execution by spawning a bare .js file.

Review habit: if a package only becomes complete during CI, review the CI assembly step as carefully as the manifest.
