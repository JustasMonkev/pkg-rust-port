Reviewing native npm CLI releases

A short lesson using this repo's @jm-pkg-rust/pkg-rust packaging work as the example.

What you will be able to do:

Given an npm CLI wrapper that delegates to native binaries, you will be able to trace the install path and spot release-only bugs that manifest checks alone miss.

1. The root package is a launcher, not the payload

In this repo, the root package package.json declares a command:

"bin": {
  "pkg": "./bin/pkg.js"
}

That means global npm installs expose the JavaScript launcher as the user-facing pkg command. The actual Rust executable lives in one of the platform packages listed under optionalDependencies, such as @jm-pkg-rust/pkg-rust-darwin-arm64 or @jm-pkg-rust/pkg-rust-linux-x64-gnu.

The npm docs make name and version central for publishable packages, describe bin as package command exposure, and document optionalDependencies, os, and cpu as package selection metadata. See npm package.json docs.

Review rule: for native npm CLIs, ask two questions: "What command did npm install?" and "Where does that command find the real binary?"

2. The launcher chooses the native package

The local launcher at bin/pkg.js builds a key from process.platform, process.arch, and, on Linux, libc. It then resolves the matching native package's package.json and executes bin/pkg or bin/pkg.exe.

darwin arm64  -> @jm-pkg-rust/pkg-rust-darwin-arm64
linux x64 musl -> @jm-pkg-rust/pkg-rust-linux-x64-musl
win32 x64 -> @jm-pkg-rust/pkg-rust-win32-x64-msvc

The execution detail matters because the launcher uses Node's spawnSync. Node documents spawnSync(command, args, options) as direct synchronous process creation. If the native file is not executable on macOS/Linux, the command fails before the Rust CLI gets control.

3. The release workflow is part of the package

The first review bug came from a gap between two true statements:

  • The build job ran chmod 755 before uploading the binary artifact.
  • The publish job downloaded that artifact and published the native package.

The missing fact was from the GitHub upload-artifact docs: artifact upload/download does not preserve Unix file modes; files can come back as 0644. A 0644 native binary inside npm's tarball will install, but the launcher will fail with EACCES.

The durable lesson: a native package's final correctness is not visible in the checked-in manifest. It is determined after CI downloads artifacts, restores permissions, and runs npm pack.

4. The fixed review pattern

The release job now performs the check at the point where it matters: after artifact download and before publish.

if [[ "${{ matrix.package }}" == win32-* ]]; then
  test -f "npm/${{ matrix.package }}/bin/pkg.exe"
else
  chmod 755 "npm/${{ matrix.package }}/bin/pkg"
  test -x "npm/${{ matrix.package }}/bin/pkg"
fi
npm pack "npm/${{ matrix.package }}" --dry-run --json

This pattern makes the release workflow prove the native package is assembled correctly. It does not just trust earlier build steps.
